# AI Agent Instructions for hierarchical-cache-simulation

## Role
You are an Expert Systems Architect and Python Developer working on a Trace-Driven CDN Hierarchical Cache Simulator. 

## Architectural Rules
1. **Strict Separation of Concerns:** This is not a monolithic script. Respect the layer boundaries.
2. **Data Layer (`src/simulator/data_access/`):** Strictly lazy-loaded generators. Never load entire logs into memory.
3. **Models Layer (`src/simulator/models/`):** Pure stateful cache physics. No file I/O. Strictly byte-aware LRU eviction.
4. **Engine Layer (`src/simulator/engine/`):** The orchestrator. Moves data between Data, Edge, and Parent.
5. **No Hallucinated Dependencies:** Rely on the Python standard library (`dataclasses`, `collections`, `typing`) unless strictly necessary.

## Domain Physics
* **Hierarchy:** Client -> Edge Cache -> Parent Cache -> Origin.
* **Eviction:** Based on `file-size` bytes, not item counts. 
* **Mutability:** The `file-size` for a specific `cachekey` can change mid-simulation. Cache models must handle byte-deltas on updates.

## Coding Philosophy
1. **Minimalism:** Write the absolute MINIMAL code required to achieve the goal. 
2. **No Over-engineering:** Strictly avoid bloated fail checks, excessive try-except ladders, or unnecessary padding. Trust the architecture.
3. **Robust, Black-Box Testing:** Tests must validate the core domain physics and edge cases. Do not write tests catered to simply pass the existing implementation. Test the expected behavior, not the internal code paths.
4. **Execution Environment:** Always assume and enforce that code, tests, and scripts are executed within the activated Python virtual environment (`venv`). Activate the environment using `source venv/bin/activate` before running any scripts or tests to ensure dependencies are correctly resolved.

## Experiment and Output Management
1. **Output Directory:** The root directory must remain clean. All outputs generated by experiments (plots, CSVs, logs, JSON results) MUST be saved in `experiments/<experiment_name>/`.
2. **Parameterized Naming:** Output files or run-specific subdirectories must include the major configuration parameters in their names to prevent overwriting and ensure traceability. 
   * *Bad:* `results.csv`, `plot.png`
   * *Good:* `results_edge_10MB_parent_50MB.csv`, `popularity_trace_small.png`
   * *For large parameter sets:* Isolate the run in a parameterized folder (e.g., `experiments/size_sweep/edge_10_parent_50/`).
3. **Automatic Scaffolding:** Scripts must use `os.makedirs(..., exist_ok=True)` to ensure the target output directories exist before attempting to save files.
4. **Always Save Raw Data:** Never generate a plot (`.png`, `.pdf`, etc.) without also saving the raw data (as `.csv` or `.json`) used to generate that plot in the same directory. Visualizations are ephemeral; the underlying data is permanent.

## Standard Operating Procedures
1. **Structured Logging:** Strictly avoid naked `print()` statements for long-running scripts. Use the `logging` module. Configure it to output to both the console and a `run.log` file saved inside the specific `experiments/<experiment_name>/` directory.
2. **Configuration Tracking:** Do not hardcode experimental parameters (cache sizes, file paths) in production scripts. Use `argparse` or config files. ALWAYS dump a `config_used.json` file into the experiment output directory so the run is permanently reproducible.
3. **Deterministic Execution:** The simulator must remain mathematically reproducible. If any stochastic or random logic is introduced, a fixed random seed (e.g., `42`) must be set globally at the start of the script.
4. **Git Hygiene:** Massive trace files and experiment outputs must never be committed. Ensure `data/`, `experiments/`, and `venv/` are strictly ignored in `.gitignore`.
